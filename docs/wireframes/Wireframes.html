Design principles guiding these wireframes

Four beliefs that push every screen away from the generic "Vaadin CRUD" starting point. Each variant below exercises one or more of these.

01

The collection is the identity

Gear isn't a list, it's who someone is. Every user-facing screen leads with the gear, not the person.

02

Connections = knowledge transfer

"Maria also owns it" is weak. "Maria wrote an ambient patch for it" is strong. Surface the knowledge, not the ownership.

03

Empty state is the marketing page

For guests and first-runs, the product itself sells the value. No separate landing page, no blank-state icons.

04

Lumo is skeleton, not skin

Vaadin gives us the bones: forms, dialogs, grids, tabs. Beyond that we break out of the 4-card dashboard reflex.

The original dashboard is four identical "view all" cards. That's a structural dead-end — it treats inventory, catalog, discussions, and people as peers, when really the user's gear should anchor the page. These four directions each pick a different anchor.

Onboarding & Guest — empty state is the marketing

The current auth gate dumps guests onto an empty dashboard with zero explanation. These three flows each remove the classic "upload then empty" trough. All are Lumo-compatible; the flow-picker is a stepper + multi-select.

01

"Pick your rack" before signup

value before commitment guest → new user

Guest sees a catalog tile-picker first. Tap 3+ pieces → the app fills with connections, threads, activity immediately. Only then does it ask for an account, to save it. Standard SaaS trick, under-used in community apps.

IA notes & open questions

Things to decide before investing in final visual design. None of these block wireframes but they'll shape the real build.

Q1

"Gear Connections" as a top-level page?

If match + knowledge live on each user's profile (V2/V3 above) + surfaces on every piece of gear, the dedicated page may be redundant. Consider demoting to a profile-sub-tab.

Q2

Inventory vs. Profile

Right now Inventory (private edit) and Profile > My Gear (public) are two pages for the same data. Propose: one inventory page with a toggle "what others see".

Q3

Patches / settings as a first-class object?

Dashboard V1 & Profile V2 assume "patches" exist — a short snippet attached to a piece of gear. If we commit to that, it unlocks a lot (see V3 taste match).

Q4

How we source gear imagery

User-uploaded photos are messy. Catalog-level photos (one per model, curated) + user photos as secondary would balance consistency and personality. Needed for Dashboards V2/V3.

Suggested next step

Pick one dashboard direction + one profile direction from above. I'll then produce a mid-fi pass of those two screens plus the connections page, still in black-and-white but with real Lumo component density — so we can de-risk before committing to a colour + type system.
