# Use Case: Register Account

## Overview

**Use Case ID:** UC-001
**Use Case Name:** Register Account
**Primary Actor:** Visitor
**Goal:** Create a new user account to access the platform's community and inventory features
**Status:** Approved
**Requirements:** FR-014

## Preconditions

- The visitor is not currently logged in.
- The visitor has access to the registration page.

## Main Success Scenario

1. Visitor navigates to the registration page.
2. System displays the registration form with fields for username, email address, and password.
3. Visitor enters a unique username, valid email address, and a password.
4. Visitor submits the form.
5. System validates that the username and email address are not already in use and that the password meets the minimum requirements.
6. System creates the account, hashing the password using bcrypt (cost factor ≥ 12).
7. System logs the visitor in and redirects them to their (empty) gear inventory.

## Alternative Flows

### A1: Username Already Taken

**Trigger:** The entered username is already registered (step 5)
**Flow:**

1. System displays an error indicating the username is unavailable.
2. Visitor enters a different username.
3. Use case continues at step 4.

### A2: Email Already Registered

**Trigger:** The entered email address is already associated with an account (step 5)
**Flow:**

1. System displays an error indicating the email is already in use.
2. Visitor enters a different email address or navigates to the log-in page.
3. Use case continues at step 4 or ends.

### A3: Password Does Not Meet Requirements

**Trigger:** The entered password fails validation (step 5)
**Flow:**

1. System highlights the password field and displays the requirements.
2. Visitor enters a compliant password.
3. Use case continues at step 4.

### A4: Form Submitted Incomplete

**Trigger:** Visitor submits the form with one or more required fields empty (step 4)
**Flow:**

1. System highlights the missing fields and displays validation messages.
2. Visitor completes the missing fields.
3. Use case continues at step 4.

## Postconditions

### Success Postconditions

- A new user account exists in the system with status "Active".
- The password is stored as a bcrypt hash with cost factor ≥ 12.
- The user is authenticated and their session is established.

### Failure Postconditions

- No account is created.
- The registration form is redisplayed with relevant error messages.

## Business Rules

### BR-001: Unique Username

Usernames must be unique across all accounts (case-insensitive).

### BR-002: Unique Email

Email addresses must be unique across all accounts.

### BR-003: Password Storage

Passwords must be hashed using bcrypt with a minimum cost factor of 12 (NFR-005).

### BR-004: Password Minimum Length

Passwords must be at least 8 characters long.
