# Use Case: Edit Gear Item

## Overview

**Use Case ID:** UC-010
**Use Case Name:** Edit Gear Item
**Primary Actor:** Home Studio Owner
**Goal:** Update the details of an existing gear item to keep the inventory accurate
**Status:** Approved
**Requirements:** FR-002, FR-005

## Preconditions

- The user is logged in.
- The gear item to be edited belongs to the user's inventory.

## Main Success Scenario

1. User opens their gear inventory (UC-012) and selects a gear item to edit.
2. System displays the edit form pre-populated with the current item details (equipment name, category, personal note).
3. User modifies one or more fields.
4. User saves the changes.
5. System validates the submission.
6. System persists the updated gear item details.
7. System redisplays the inventory reflecting the changes.

## Alternative Flows

### A1: Validation Error

**Trigger:** A submitted field violates a constraint (step 5)
**Flow:**

1. System highlights the offending field(s) and displays an error message.
2. User corrects the value.
3. Use case continues at step 4.

### A2: User Cancels

**Trigger:** User discards changes before saving (step 3 or 4)
**Flow:**

1. System discards unsaved changes.
2. Use case ends with no modification to the gear item.

## Postconditions

### Success Postconditions

- The gear item is updated in the user's inventory with the new details.

### Failure Postconditions

- The gear item is unchanged.
- The edit form is redisplayed with validation errors.

## Business Rules

### BR-001: Ownership Enforcement

A user may only edit gear items that belong to their own inventory.

### BR-002: Category Required

The category field cannot be cleared; a valid category must always be assigned.
