# Use Case: Log In

## Overview

**Use Case ID:** UC-002
**Use Case Name:** Log In
**Primary Actor:** Visitor
**Goal:** Authenticate with the platform to access account-specific features
**Status:** Approved
**Requirements:** FR-015

## Preconditions

- The visitor is not currently logged in.
- A registered account exists for the visitor.

## Main Success Scenario

1. Visitor navigates to the log-in page.
2. System displays the log-in form with fields for email address and password.
3. Visitor enters their email address and password.
4. Visitor submits the form.
5. System validates the credentials against the stored account.
6. System establishes an authenticated session and redirects the visitor to their gear inventory.

## Alternative Flows

### A1: Invalid Credentials

**Trigger:** Email address or password does not match a stored account (step 5)
**Flow:**

1. System displays a generic error message ("Invalid email or password") without indicating which field is incorrect.
2. Visitor re-enters credentials.
3. Use case continues at step 4.

### A2: Form Submitted Incomplete

**Trigger:** Visitor submits the form with one or more fields empty (step 4)
**Flow:**

1. System highlights the missing fields.
2. Visitor completes the missing fields.
3. Use case continues at step 4.

## Postconditions

### Success Postconditions

- An authenticated session is established for the user.
- The session expires after 30 minutes of inactivity (NFR-006).

### Failure Postconditions

- No session is created.
- The log-in form is redisplayed with an error message.

## Business Rules

### BR-001: Generic Error Message

The system must not disclose whether the email or the password was incorrect, to prevent account enumeration.

### BR-002: Session Expiry

Authenticated sessions expire after 30 minutes of inactivity (NFR-006).
